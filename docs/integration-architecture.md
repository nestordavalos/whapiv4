## Startup Sequence

When `server.ts` runs, the following initializes in order:

| Step | Module | Action |
|------|--------|--------|
| 1 | `bootstrap.ts` | Load `.env` via dotenv |
| 2 | `app.ts` | Initialize Express with full middleware chain |
| 3 | Swagger | Mount Swagger UI at `/api-docs` |
| 4 | HTTP | `app.listen(PORT)` — start HTTP server |
| 5 | Cron | Register `ClosedAllOpenTickets()` — runs every 1 minute |
| 6 | Storage Sync | If S3 with fallback enabled, start `StorageSyncService.startAutoSync(5min)` |
| 7 | Socket.IO | `initIO(server)` — bind Socket.IO to HTTP server |
| 8 | WhatsApp | `StartAllWhatsAppsSessions()` — init all WhatsApp connections sequentially |

### WhatsApp Session Boot

`StartAllWhatsAppsSessions()` queries all WhatsApp connections from DB and for each:

1. Updates status to `"OPENING"`, emits `whatsappSession` socket event
2. `initWbot(whatsapp)` — creates whatsapp-web.js Client with Puppeteer
3. `wbotMessageListener(wbot)` — registers message event handlers
4. `wbotMonitor(wbot, whatsapp)` — registers status/disconnect handlers

## Socket.IO Architecture

### Connection & Authentication

On each connection (`socket.ts`):
1. Extract `token` from `socket.handshake.query`
2. Verify JWT → `{ id, username, profile }`
3. Look up user in DB; disconnect if not found
4. Mark `user.online = true`, call `updateActivity(userId)`

### Multi-Tab Connection Tracking

A `Map<number, number>` tracks active connections per userId. On disconnect, user is only marked offline when count drops to zero.

### Room System

| Client Event | Room Joined | Purpose |
|-------------|-------------|---------|
| (auto on connect) | `userId` (string) | Personal channel |
| `joinChatBox` | `ticketId` | Real-time chat for specific ticket |
| `joinNotification` | `"notification"` | Global notification channel |
| `joinTickets` | `status` (e.g. "open") | Ticket list updates by status |

### Server-Emitted Events

| Event | Source | Payload |
|-------|--------|---------|
| `whatsappSession` | wbot.ts, wbotMonitor.ts | `{ action: "update", session }` |
| `ticket` | UpdateTicketService, wbotCloseTickets | `{ action: "update"/"delete", ticket }` |
| `appMessage` | wbotMessageListener | `{ action: "create"/"update", message, ticket, contact }` |
| `userStatus` | socket.ts | `{ userId, status }` |
| `whatsapp:error` | SendWhatsAppMessage | `{ whatsappId, message }` |
| `session:expired` | isAuth middleware | `{ userId }` |
| `ticketCounts` | (various) | Tab badge counts |

### Session Manager

`sessionManager.ts` tracks user activity:
- `INACTIVITY_LIMIT_MS = 8 hours`
- Updated on each socket connection and on each socket event (`socket.onAny`)
- Functions: `updateActivity()`, `getLastActivity()`, `isExpired()`, `clearSession()`

## WhatsApp Integration

### Session Lifecycle

```
OPENING → qr (show QR) → authenticated → ready (CONNECTED)
                                              │
                                              ▼
                                     [Active Session]
                                     ├── Receives messages
                                     ├── Sends messages
                                     ├── Ping health check (60s)
                                     └── syncUnreadMessages()
                                              │
                              ┌───────────────┼───────────────┐
                              ▼               ▼               ▼
                        auth_failure    disconnected    manual shutdown
                              │               │               │
                              ▼               ▼               ▼
                        DISCONNECTED    DISCONNECTED    DISCONNECTED
                        (retry ≤5)      (auto-reconnect)  (cleanup files)
```

### Client Configuration

- **Library**: whatsapp-web.js (Puppeteer-based)
- **Auth Strategy**: `LocalAuth({ clientId: "bd_{whatsappId}" })`
- **Chrome**: headless via `CHROME_BIN` or Browserless WebSocket (`CHROME_WS`)
- **Health Check**: `setInterval(60s)` calls `wbot.getState()`

### Session Management Functions

|----------|------|--------|

## Message Flow

### Incoming Message

```
WhatsApp Server
    │
    ▼
wbot event: message_create / media_uploaded
    │
    ▼
handleMessage(msg, wbot)
    │
    ├── isValidMsg()? → filter by type (chat, audio, image, video, document, vcard, location)
    ├── Ignore: own messages with \u200e, status@broadcast, groups (if disabled)
    ├── Duplicate check: Message.findByPk(msg.id.id) → update ack only if exists
    │
    ├── verifyContact() → CreateOrUpdateContactService
    ├── FindOrCreateTicketService() → reuse open/pending or create new
    │
    ├── [has media?] → msg.downloadMedia() → verifyMediaMessage()
    │                    → StorageService.uploadBase64() → save to S3/local
    │   [no media]  → verifyMessage() → CreateMessageService
    │
    ├── Webhook: sendMessageReceivedWebhook() / sendMessageSentWebhook()
    │
    ├── [no queue?] → verifyQueue() → show queue selection menu
    ├── [has integration?] → typebotListener() → Typebot conversation
    ├── [has queue, no user?] → sayChatbot() → internal chatbot menu
    │
    └── Socket emit: "appMessage" { action: "create", message, ticket, contact }
```

### Outgoing Message

| Method | Service | Mechanism |
|--------|---------|-----------|
| Text | `SendWhatsAppMessage.ts` | Retry with exponential backoff (3 attempts, 500ms base) + jitter |
| Media (file) | `SendWhatsAppMedia.ts` | `MessageMedia.fromFilePath()` with caption |
| Media (URL) | `SendWhatsAppMediaFromUrl.ts` | `MessageMedia.fromUrl()` with fallback |
| Media (base64) | `SendWhatsAppMediaFromBase64.ts` | Direct base64 to WhatsApp |
| Forward | `ForwardWhatsAppMessage.ts` | Message forwarding |
| Edit | `EditWhatsAppMessage.ts` | Edit sent messages |
| Delete | `DeleteWhatsAppMessage.ts` | Delete messages |

### Message ACK Flow

```
message_ack event → handleMsgAck(msg, ack)
    │
    ├── Update Message.ack in DB
    ├── Emit socket: "appMessage" { action: "update", message }
    └── Webhook: sendMessageAckWebhook()
```

ACK levels: 0=PENDING, 1=SENT, 2=RECEIVED, 3=READ, 4=PLAYED

### Other Message Events

| Event | Handler | Action |
|-------|---------|--------|
| `message_revoke_everyone` | Direct | Mark `isDeleted: true` in DB |
| `message_edit` | `HandleMessageEditService` | Update body, emit socket |

## Ticket Lifecycle

### States

| State | Description | Bot Active |
|-------|-------------|------------|
| `pending` | New, unassigned ticket | Yes (`isBot=true`) |
| `open` | Assigned to agent, active conversation | No |
| `closed` | Conversation finished | No |
| `nps` | Awaiting customer rating (NPS) | No |

### State Transitions

```
                         ┌──────────────────────────────────┐
                         │                                  │
[New message] ──► pending ──(agent accepts)──► open ──(close)──► closed
                    │                           │              ▲
                    │                           │              │
                    │                    (close+useNPS)──► nps ┘
                    │                                     (rating or 10min timeout)
                    │
                    └──(chatbot assigns userId)──► open
                    
[Contact sends "#"] ──► Reset queue/user ──► pending (show menu again)

[New msg from contact on closed ticket within timeCreateNewTicket] ──► pending
```

### Automated Actions

| Trigger | Action | Configuration |
|---------|--------|---------------|
| Queue assigned | Send greeting message | `Queue.greetingMessage` |
| Queue menu shown | Send numbered options | Automatic from queue list |
| Outside work hours | Send absence message | `Queue.absenceMessage` + day/hour schedule |
| Ticket closed (with NPS) | Send rating request | `Setting.ratingMessage` |
| After NPS received | Send farewell | `Whatsapp.farewellMessage` |
| Inactivity timeout | Send inactive message + close | `Whatsapp.inactiveMessage` + `timeInactiveMessage` |
| NPS timeout (10 min) | Auto-close | Cron job |
| Contact sends "#" | Reset to queue menu | Built-in keyword |

### Key Services

| Service | Purpose |
|---------|---------|
| `CreateTicketService` | Create ticket with relations |
| `FindOrCreateTicketService` | Reuse open/pending tickets, respect `timeCreateNewTicket` |
| `UpdateTicketService` | Handle transitions, tracking, sockets, webhooks, rating |
| `FindOrCreateATicketTrakingService` | Maintain `TicketTraking` timestamps (queued, started, rated, closed) |

## Webhook System

### Configuration

Each WhatsApp connection has webhook settings:
- `webhookEnabled: boolean` — global toggle
- `webhookUrls: WebhookConfig[]` — array of webhook destinations

```typescript
interface WebhookConfig {
  id: string;       // UUID
  name: string;     // Display name
  url: string;      // Destination URL
  enabled: boolean; // Per-webhook toggle
  events: WebhookEventType[];  // Event filter
}
```

### Event Types

| Event | Triggered When |
|-------|---------------|
| `message_received` | Incoming message from contact |
| `message_sent` | Outgoing message from agent/bot |
| `message_ack` | Message delivery/read status change |
| `connection_update` | WhatsApp connection status change |
| `ticket_created` | New ticket created |
| `ticket_updated` | Ticket status/assignment changed |
| `ticket_closed` | Ticket closed |
| `contact_created` | New contact discovered |
| `contact_updated` | Contact info updated |

### Dispatch Flow

```
Event occurs → SendWebhookEvent({ whatsappId, event, data })
    │
    ├── Get config from cache (TTL 60s) or query DB
    ├── Build payload: { event, timestamp, connectionId, connectionName, connectionNumber, data }
    │
    └── For each webhook where enabled=true AND event in events filter:
            │
            └── HTTP POST (fire-and-forget, async)
                Headers: Content-Type, X-Webhook-Event, X-Webhook-Name, X-Connection-Id
                Timeout: 10 seconds
                On error: log + Sentry.captureException()
```

### Important Notes

- **No retry logic** — webhooks are sent once. On failure, errors are logged and reported to Sentry.
- **No queue** — dispatch is fire-and-forget via async axios call.
- **Cache** — webhook config cached per whatsappId with 60-second TTL. Cache invalidated on config update.

## Media Flow

### Incoming Media (WhatsApp → Storage)

```
msg.downloadMedia() → base64 data
    │
    ▼
StorageService.uploadBase64(data, filename, mimetype)
    │
    ├── [S3 primary] → S3StorageProvider.uploadBuffer()
    │       ├── Success → optionally delete local copy
    │       └── Fail + fallback → LocalStorageProvider + PendingUpload record
    │
    └── [Local primary] → LocalStorageProvider → write to ./public/{filename}
    
    → Message record: mediaUrl=filename, mediaType=image|audio|video|document
```

### Outgoing Media (User Upload → WhatsApp)

```
HTTP POST multipart → multer → ./public/{timestamp}.{ext}
    │
    ▼
SendWhatsAppMedia: MessageMedia.fromFilePath(media.path)
    │
    └── wbot.sendMessage(chatId, media, { caption })
```

### Storage Configuration

| Type | Description | Config |
|------|-------------|--------|
| `local` | Files in `./public/`, served via Express static | Default |
| `s3` | AWS S3 bucket | Requires: bucket, region, access key, secret key |
| `s3_compatible` | MinIO, DO Spaces, Cloudflare R2 | Requires: endpoint + `FORCE_PATH_STYLE` for MinIO |

### Serving

- **Local files**: Express static mount at `/public`
- **S3 files**: `StorageService.getResolvedPublicUrl(filename)` returns S3 URL or CDN URL
- **Fallback**: if S3 unreachable, serve from local copy if available

## Background Jobs

### Cron: Close Inactive Tickets

`server.ts` → `cron.schedule("*/1 * * * *")`

Runs `ClosedAllOpenTickets()` every minute:
- Processes open tickets in batches of 200
- Closes tickets where NPS rating timeout (10 min) exceeded
- Closes tickets where inactivity timeout exceeded (per WhatsApp connection setting)
- Emits socket `ticket` delete events for each closed ticket

### Timer: Storage Sync

`StorageSyncService.startAutoSync(5 * 60 * 1000)` — every 5 minutes:
- Processes `PendingUpload` records with status `pending` or `failed`
- Attempts S3 upload from local copy
- On success with `deleteLocalAfterUpload=true`, removes local file

### Timer: WhatsApp Health Check

Per-session `setInterval(60000)`:
- Calls `wbot.getState()` to verify session alive
- Clears interval if session terminated

> **Note**: The project does **not** use Bull, BullMQ, Redis, or any job queue system. All background work runs via cron and `setInterval`.
